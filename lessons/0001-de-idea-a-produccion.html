Lección 0001 · 8 minutos

Un cambio, de idea a producción

El ciclo no es una colección de herramientas. Es una cadena de evidencia: cada gate reduce una incertidumbre y el mismo artefacto avanza hasta producción.

●

Realidad del repositorio: commit stage y artifact acceptance ya están verdes. Preview, promotion y recovery son el próximo incremento; aparecen marcados como “siguiente”.

Tu victoria en esta lección

Poder narrar el recorrido sin nombrar YAML

Seguir el cambio

Identificar qué entra y qué sale de cada stage.

Ubicar la evidencia

Explicar qué incertidumbre elimina cada gate.

Detectar el anti-patrón

Reconocer por qué recompilar antes de producción rompe la cadena.

Tres términos que no son sinónimos

CI, delivery y deployment

Continuous Integration

Integramos cambios pequeños con frecuencia y obtenemos feedback automático. Una falla detiene el trabajo normal hasta recuperar `main`.

Continuous Delivery

Cada cambio verde queda desplegable. Producción puede requerir una decisión explícita, pero no trabajo técnico adicional.

Continuous Deployment

Cada cambio verde en `main` avanza automáticamente a producción. Este es nuestro estado objetivo.

01 · Intake

Convertir una señal en trabajo trazable

Una necesidad entra como GitHub Issue. Triage separa bug de enhancement y decide si falta información, necesita una persona o ya puede tomarla un agente.

Entrada
Problema u oportunidad
Evidencia
Impacto y contexto suficientes
Salida
Issue con estado canónico

02 · Shape

Reducir ambigüedad antes de producir código

`/grill-with-docs` afila decisiones, `/to-spec` captura el comportamiento esperado y `/to-tickets` recorta tracer bullets verticales con dependencias explícitas.

Entrada
Issue aceptado
Evidencia
Seams y criterios observables
Salida
Tickets pequeños y ordenados

03 · Implement

Hacer el cambio seguro más pequeño

Una rama muy corta toma un ticket. Para comportamiento usamos Red → Green; cada commit es atómico, tiene Conventional Commit con scope y queda verde bajo el feedback relevante.

Entrada
Un tracer bullet
Evidencia
Test rojo y luego verde
Salida
PR pequeña y revertible

04 · Commit stage

Responder rápido: “¿vale la pena seguir?”

GitHub Actions valida commits, formato, lint, tipos y unit tests. Luego compila una sola vez, crea Vercel Build Output y empaqueta `release-<SHA>` con checksum. La corrida observada tarda unos 22 segundos.

Entrada
Commit identificado por SHA
Evidencia
Checks baratos y deterministas
Salida
`release.tgz` inmutable

05 · Artifact acceptance

Probar lo que realmente se desplegará

Otro job descarga el artefacto, verifica SHA-256, lo sirve por HTTP y Playwright observa la interfaz pública. No consulta internals ni vuelve a ejecutar el build. La corrida observada tarda unos 52 segundos.

Entrada
Artefacto + checksum
Evidencia
Comportamiento en navegador
Salida
Candidato apto para deploy

06 · Preview

Ejecutar el candidato en infraestructura real

El próximo incremento desplegará el mismo Vercel Build Output con `vercel deploy --prebuilt`. Un smoke remoto comprobará URL, contenido y SHA antes de permitir promoción.

Entrada
Artefacto aceptado
Evidencia
Smoke contra URL preview
Salida
Deployment inmutable verificado

07 · Promote

Cambiar el puntero, no los bytes

En `main`, `vercel promote <preview-url> --yes` hará actual ese deployment. Promover evita la divergencia “funcionó en staging, falló en producción” causada por un segundo build.

Entrada
Preview verificado
Evidencia
Todos los gates verdes
Salida
Mismos bytes en producción

08 · Operate and learn

Verificar el resultado y cerrar el loop

Un smoke de producción comprueba el SHA esperado. Si falla, se detiene el flujo y se promueve un deployment conocido como bueno. Lead time, frecuencia, fallas y recovery muestran dónde mejorar el sistema después.

Entrada
Deployment promovido
Evidencia
Salud + métricas de entrega
Salida
Aprendizaje o recuperación

La idea que sostiene todo

Build once, promote

source@SHA → artifact(SHA) → acceptance → preview → production

Identidad

El SHA conecta código, artefacto, deployment y diagnóstico.

Inmutabilidad

Después del build, ningún stage modifica los bytes.

Monotonicidad

Cada gate suma evidencia; una falla detiene la promoción.

Estado honesto

Tenemos una tubería parcial, no CD completo
